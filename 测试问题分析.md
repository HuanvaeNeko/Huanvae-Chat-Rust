# 测试脚本问题分析

## 🔍 问题描述

在第 16-17 步测试"删除设备后 Token 是否失效"时，发现 Token 仍然有效。

## 📊 测试输出分析

### 第 15 步：查看设备列表
```json
{
  "devices": [
    {
      "device_id": "ceff28ac-d9ed-4ca9-87d1-5e7724bc48b5",
      "device_info": "Linux/TestScript",
      "is_current": false,        // ← 旧设备（第一次登录）
      "created_at": "2025-11-25T11:53:52.318162"
    },
    {
      "device_id": "2613f590-084c-4c3d-ba71-022926d6e6aa",
      "device_info": "Linux/TestScript",
      "is_current": true,         // ← 当前设备（修改密码后重新登录）
      "created_at": "2025-11-25T11:54:06.817117"
    }
  ],
  "total": 2
}
```

### 第 16 步：删除设备
- **测试脚本删除的设备**: `ceff28ac-d9ed-4ca9-87d1-5e7724bc48b5` (旧设备)
- **当前使用的 Token 所属设备**: `2613f590-084c-4c3d-ba71-022926d6e6aa` (当前设备)

### 第 17 步：测试结果
- ✗ Token 仍然有效（可能存在问题）← **这个判断错误！**

## ⚠️ 问题根源

### 测试脚本的逻辑错误

测试脚本从设备列表中取了**第一个设备**（`devices[0]`）来删除：
```bash
DEVICE_ID=$(echo "$DEVICES" | jq -r '.devices[0].device_id' 2>/dev/null)
```

但**第一个设备不是当前使用的设备**！

- 设备列表按 `last_used_at DESC` 排序
- `devices[0]` 是旧设备（`is_current: false`）
- 当前 Token 属于 `devices[1]`（`is_current: true`）

### 系统的正确行为

根据 `src/auth/handlers/revoke_device.rs` 的代码：

```rust
// 撤销指定设备（远程登出）
pub async fn revoke_device_handler(...) {
    // 1. 启用黑名单检查（窗口 15 分钟）
    state.blacklist_service.enable_blacklist_check(&auth_context.user_id).await?;

    // 2. 按设备读取缓存并批量拉黑 Access Token
    let cached = state.device_service.list_cached_access_tokens(&auth_context.user_id, &device_id).await?;
    for (jti, exp) in cached.iter() {
        state.blacklist_service.add_to_blacklist(jti, ...).await?;
    }

    // 3. 如果缓存为空且删除的是当前设备，兜底处理
    if cached.is_empty() && device_id == auth_context.device_id {
        state.blacklist_service.add_to_blacklist(&auth_context.claims.jti, ...).await?;
    }

    // 4. 撤销设备（Refresh Token）
    state.device_service.revoke_device(&auth_context.user_id, &device_id).await?;
}
```

**关键点**：
- 只会将**被删除设备的 Access Token** 加入黑名单
- 如果删除的是**旧设备**，当前设备的 Token **不会**受影响
- 只有删除**当前设备**时，当前 Token 才会失效

## ✅ 正确的测试逻辑

### 方案 1：删除当前设备

```bash
# 从设备列表中找到 is_current = true 的设备
CURRENT_DEVICE_ID=$(echo "$DEVICES" | jq -r '.devices[] | select(.is_current == true) | .device_id')

# 删除当前设备
DELETE_DEVICE=$(api_call DELETE "/api/auth/devices/${CURRENT_DEVICE_ID}" "$USER1_TOKEN")

# 此时当前 Token 应该失效
```

### 方案 2：两阶段测试

```bash
# 阶段 1: 删除旧设备（不应影响当前 Token）
OLD_DEVICE_ID=$(echo "$DEVICES" | jq -r '.devices[] | select(.is_current == false) | .device_id')
api_call DELETE "/api/auth/devices/${OLD_DEVICE_ID}" "$USER1_TOKEN"

# 验证当前 Token 仍然有效（预期行为）
PROFILE=$(api_call GET /api/profile "$USER1_TOKEN")
if echo "$PROFILE" | jq -e '.data' > /dev/null 2>&1; then
    log_success "✓ 删除旧设备后，当前 Token 仍然有效（预期行为）"
fi

# 阶段 2: 删除当前设备（应使 Token 失效）
CURRENT_DEVICE_ID=$(echo "$DEVICES" | jq -r '.devices[] | select(.is_current == true) | .device_id')
api_call DELETE "/api/auth/devices/${CURRENT_DEVICE_ID}" "$USER1_TOKEN"

# 验证当前 Token 已失效（预期行为）
PROFILE_AFTER=$(api_call GET /api/profile "$USER1_TOKEN")
if echo "$PROFILE_AFTER" | grep -qi "unauthorized\|revoked\|401"; then
    log_success "✓ 删除当前设备后，Token 已失效（预期行为）"
fi
```

## 📋 为什么会有两个设备？

回溯测试流程：

1. **第 2 步**：用户1 第一次登录 → 创建设备1
2. **第 12 步**：用户1 修改密码
3. **第 13 步**：用户1 用新密码重新登录 → 创建设备2
4. **第 15 步**：查看设备列表 → 显示 2 个设备

所以：
- 设备1 = 第一次登录的设备（旧 Token）
- 设备2 = 修改密码后重新登录的设备（当前 Token）

## 🎯 结论

**系统行为完全正确！测试脚本的判断逻辑有误。**

- ✅ 删除旧设备，当前 Token **应该**保持有效
- ✅ 删除当前设备，当前 Token **才会**失效

### 2. 更新相关文档

> **撤销设备行为说明**：
> 
> - 删除指定设备会撤销该设备的 Refresh Token
> - 删除指定设备会将该设备的所有 Access Token 加入黑名单
> - **仅当删除的是发起请求的当前设备时**，当前 Access Token 才会失效
> - 删除其他设备不会影响当前设备的 Token

## 📝 测试脚本修正代码

```bash
log_step "第 16 步：删除旧设备（应不影响当前 Token）"

OLD_DEVICE_ID=$(echo "$DEVICES" | jq -r '.devices[] | select(.is_current == false) | .device_id' | head -1)
if [ -n "$OLD_DEVICE_ID" ] && [ "$OLD_DEVICE_ID" != "null" ]; then
    DELETE_DEVICE=$(api_call DELETE "/api/auth/devices/${OLD_DEVICE_ID}" "$USER1_TOKEN")
    log_info "删除旧设备: $OLD_DEVICE_ID"
    
    # 验证当前 Token 仍然有效
    PROFILE_CHECK=$(api_call GET /api/profile "$USER1_TOKEN")
    if echo "$PROFILE_CHECK" | jq -e '.data' > /dev/null 2>&1; then
        log_success "✓ 删除旧设备后，当前 Token 仍然有效（预期行为）"
    else
        log_error "✗ 当前 Token 意外失效"
    fi
fi

log_step "第 17 步：删除当前设备（应使 Token 失效）"

CURRENT_DEVICE_ID=$(echo "$DEVICES" | jq -r '.devices[] | select(.is_current == true) | .device_id')
if [ -n "$CURRENT_DEVICE_ID" ] && [ "$CURRENT_DEVICE_ID" != "null" ]; then
    DELETE_CURRENT=$(api_call DELETE "/api/auth/devices/${CURRENT_DEVICE_ID}" "$USER1_TOKEN")
    log_info "删除当前设备: $CURRENT_DEVICE_ID"
    
    sleep 2  # 等待黑名单同步
    
    # 验证当前 Token 已失效
    PROFILE_AFTER=$(api_call GET /api/profile "$USER1_TOKEN")
    if echo "$PROFILE_AFTER" | grep -qi "unauthorized\|revoked\|401\|无效\|失效"; then
        log_success "✓ 删除当前设备后，Token 已失效（预期行为）"
    else
        log_error "✗ Token 仍然有效（不符合预期）"
    fi
fi
```
