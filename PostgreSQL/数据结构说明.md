## 🏗️ 架构设计原则

### 1. 数据规范化
- ✅ 避免在TEXT字段中存储结构化数据
- ✅ 使用外键保证数据完整性
- ✅ 为高频查询创建合适的索引

### 2. 软删除策略
- ✅ 好友关系：`status` 字段标记 active/ended
- ✅ 好友消息：独立的删除标记（双方可独立删除）
- ✅ 文件访问权限：`revoked-at` 时间戳

### 3. 性能优化
- ✅ 部分索引（Partial Index）减少索引大小
- ✅ 复合索引覆盖高频查询场景
- ✅ 时间字段统一使用UTC时间戳

### 4. 扩展性设计
- ✅ UUID作为主键支持分布式系统
- ✅ 文件去重机制支持跨用户共享
- ✅ 权限表设计支持灵活的访问控制

## user-access-cache（Access Token缓存表）

- 用途：缓存近 15 分钟签发的 Access Token，用于按设备 ID 精准拉黑。
- 触发：登录/刷新时写入；登出/删除设备时读取并批量拉黑。
- 字段：
  - `jti`：JWT 唯一标识（唯一键/主键）。
  - `user-id`：所属用户 ID（外键到 `users.user-id`）。
  - `device-id`：设备标识（与 Claims 对齐）。
  - `exp`：过期时间（Access Token 过期）。
  - `issued-at`：签发时间。
  - `inserted-at`：写入时间（默认当前时间）。
- 索引：
  - `UNIQUE(jti)`（主键）。
  - `INDEX(user-id, device-id)`（按设备查找）。
  - `INDEX(exp)`（按过期清理）。
- TTL 与清理：
  - 自动清理 `exp < now()` 或 `inserted-at < now() - 15min`。
- 读取策略：
  - 事件（登出/删除设备）读取 `user-id + device-id` 且 `exp > now()` 的所有记录，批量写入 `token-blacklist`。
- 写入策略：
  - 在 `generate_access_token`（登录与刷新）签发后写入一条，`ON CONFLICT(jti) DO NOTHING` 幂等。

## need-blacklist-check（修改说明）

- 设置为 true：
  - 在“拉黑型事件”开始阶段立即确保为 true，并将 `blacklist-check-expires-at = now + 15min`；若已为 true 则延长窗口到最新时间。
  - 事件包括：登出、删除当前设备、删除其它设备。
- 自动复位：
  - 中间件在请求时发现过期即复位；定时任务也会复位。
- 目的：
  - 窗口期内对所有受保护路由执行黑名单查询，确保被拉黑的 Access Token 立即失效。

### 黑名单检查控制字段

| 字段名 | 类型 | 说明 | 存储内容结构 |
|--------|------|------|-------------|
| need-blacklist-check | BOOLEAN DEFAULT false | 是否需要黑名单检查 | true: 需要检查<br>false: 跳过检查 |
| blacklist-check-expires-at | TIMESTAMP | 黑名单检查过期时间 | 15分钟后自动过期<br>过期后自动关闭检查 |

#### 使用说明

此字段用于智能性能优化，在以下安全事件时启用：
- 用户修改密码
- 远程登出其他设备
- 检测到 Token 泄露

**工作原理：**
1. 正常情况下 `need-blacklist-check = false`，跳过黑名单查询（高性能）
2. 安全事件发生时设置 `need-blacklist-check = true`，启用黑名单检查（15分钟）
3. 15分钟后自动恢复为 `false`（定时任务清理）

## 📊 表名: token-blacklist (Token黑名单表)

### 用途
紧急撤销 Access Token（15分钟有效期内需要立即失效的情况）

### 字段设计

| 字段名 | 类型 | 说明 | 备注 |
|--------|------|------|------|
| jti | TEXT PRIMARY KEY | Token的jti字段 | JWT中的唯一标识 |
| user-id | TEXT NOT NULL | 用户ID | 便于查询用户的所有黑名单Token |
| token-type | TEXT | Token类型 | "access" 或 "refresh" |
| expires-at | TIMESTAMP NOT NULL | Token过期时间 | 与原Token一致 |
| blacklisted-at | TIMESTAMP DEFAULT CURRENT_TIMESTAMP | 加入黑名单时间 | |
| reason | TEXT | 撤销原因 | 安全事件、密码修改等 |

### 索引

- `idx-blacklist-jti` - 主键索引
- `idx-blacklist-user-id` - 用于查询用户黑名单
- `idx-blacklist-expires` - 用于定期清理过期记录

### 清理策略

定期删除已过期的黑名单记录（因为过期的Token本身已经无效）：
DELETE FROM "token-blacklist" WHERE "expires-at" < NOW();

## 📊 表名: friend-messages (好友消息表)

### 用途
存储用户之间的所有聊天消息，支持文本、图片、视频、文件等多种消息类型。

### 字段设计

| 字段名 | 类型 | 说明 | 备注 |
|--------|------|------|------|
| message-uuid | TEXT PRIMARY KEY | 消息唯一标识 | UUID v4 |
| conversation-uuid | TEXT NOT NULL | 会话唯一标识 | 双方用户ID排序后组合 |
| sender-id | TEXT NOT NULL | 发送者ID | 外键关联 users.user-id |
| receiver-id | TEXT NOT NULL | 接收者ID | 外键关联 users.user-id |
| message-content | TEXT NOT NULL | 消息内容 | 文本消息的内容，媒体消息可为描述 |
| message-type | TEXT NOT NULL | 消息类型 | text/image/video/file |
| file-url | TEXT | 文件URL | 媒体消息时存储文件访问URL（UUID格式） |
| file-uuid | VARCHAR(36) | 文件UUID | 关联 file-uuid-mapping 表，用于权限控制 |
| file-size | BIGINT | 文件大小 | 字节为单位 |
| send-time | TIMESTAMP NOT NULL | 发送时间 | UTC时间，服务器生成 |
| is-deleted-by-sender | BOOLEAN DEFAULT false | 发送者是否删除 | 软删除标记 |
| is-deleted-by-receiver | BOOLEAN DEFAULT false | 接收者是否删除 | 软删除标记 |

### conversation-uuid 生成规则

将两个用户ID排序后组合，确保双方查询同一会话：

```rust
// 示例代码
fn generate_conversation_uuid(user_id_1: &str, user_id_2: &str) -> String {
    let mut ids = vec![user_id_1, user_id_2];
    ids.sort();
    format!("conv-{}-{}", ids[0], ids[1])
}

// 示例结果
// user-123 和 user-456 → "conv-user-123-user-456"
// user-456 和 user-123 → "conv-user-123-user-456" (相同)
```

### 索引

- `idx-messages-conversation-time` - 会话+时间复合索引（主查询）
- `idx-messages-sender` - 发送者索引
- `idx-messages-receiver` - 接收者索引
- `idx-messages-send-time` - 发送时间索引
- `idx-messages-conversation` - 会话索引
- `idx-friend-messages-file-uuid` - 文件UUID索引（仅索引有文件的消息）

### 软删除说明

- 每个用户可以独立删除消息，不影响对方查看
- 查询时需要根据当前用户过滤已删除的消息
- 示例查询：
  ```sql
  SELECT * FROM "friend-messages"
  WHERE "conversation-uuid" = $1
    AND (
      ("is-deleted-by-sender" = false AND "sender-id" = $2) OR
      ("is-deleted-by-receiver" = false AND "receiver-id" = $2)
    )
  ORDER BY "send-time" DESC
  LIMIT 50;
  ```

## 📊 表名: friend-unread-messages (未读消息表)

### 用途
存储每个用户与好友的会话信息和未读消息计数，用于会话列表展示和未读角标。

### 字段设计

| 字段名 | 类型 | 说明 | 备注 |
|--------|------|------|------|
| unread-id | TEXT PRIMARY KEY | 记录唯一标识 | UUID |
| user-id | TEXT NOT NULL | 用户ID | 外键关联 users.user-id |
| conversation-uuid | TEXT NOT NULL | 会话唯一标识 | 与消息表对应 |
| friend-id | TEXT NOT NULL | 好友ID | 用于显示好友信息 |
| unread-count | INTEGER DEFAULT 0 | 未读消息数量 | 显示角标 |
| last-message-uuid | TEXT | 最后一条消息UUID | 用于分页查询 |
| last-message-content | TEXT | 最后一条消息内容 | 会话列表预览 |
| last-message-type | TEXT | 最后一条消息类型 | text/image/video/file |
| last-message-time | TIMESTAMP | 最后消息时间 | UTC时间 |
| updated-at | TIMESTAMP | 更新时间 | 自动更新 |

### 索引

- `idx-unread-user-updated` - 用户+更新时间（会话列表查询）
- `idx-unread-conversation` - 会话索引
- `idx-unread-count` - 未读数索引（部分索引，仅索引 unread-count > 0）

### 唯一约束

- `(user-id, conversation-uuid)` - 每个用户对每个会话只有一条记录

### 更新策略

**收到新消息时:**
```sql
INSERT INTO "friend-unread-messages" (
    "unread-id", "user-id", "conversation-uuid", "friend-id",
    "unread-count", "last-message-uuid", "last-message-content",
    "last-message-type", "last-message-time"
)
VALUES ($1, $2, $3, $4, 1, $5, $6, $7, $8)
ON CONFLICT ("user-id", "conversation-uuid")
DO UPDATE SET
    "unread-count" = "friend-unread-messages"."unread-count" + 1,
    "last-message-uuid" = $5,
    "last-message-content" = $6,
    "last-message-type" = $7,
    "last-message-time" = $8,
    "updated-at" = CURRENT_TIMESTAMP;
```

**标记已读时:**
```sql
UPDATE "friend-unread-messages"
SET "unread-count" = 0,
    "updated-at" = CURRENT_TIMESTAMP
WHERE "user-id" = $1 AND "conversation-uuid" = $2;
```

### 会话列表查询示例

```sql
SELECT 
    u."unread-id",
    u."conversation-uuid",
    u."friend-id",
    usr."user-nickname" AS friend_nickname,
    usr."user-avatar-url" AS friend_avatar,
    u."unread-count",
    u."last-message-content",
    u."last-message-type",
    u."last-message-time"
FROM "friend-unread-messages" u
LEFT JOIN "users" usr ON usr."user-id" = u."friend-id"
WHERE u."user-id" = $1
ORDER BY u."last-message-time" DESC
LIMIT 50;
```

## 📊 表名: file-uuid-mapping (文件UUID映射表)

### 用途
实现文件去重的核心表，通过UUID映射实现跨用户文件共享。

### 字段设计

| 字段名 | 类型 | 说明 | 备注 |
|--------|------|------|------|
| uuid | VARCHAR(36) PRIMARY KEY | 随机UUID | 用于访问URL |
| physical-file-key | VARCHAR(500) NOT NULL | 实际物理文件路径 | MinIO中的真实存储路径 |
| file-hash | VARCHAR(64) NOT NULL | SHA-256哈希 | 用于去重查询 |
| file-size | BIGINT NOT NULL | 文件大小 | 字节为单位 |
| content-type | VARCHAR(100) NOT NULL | MIME类型 | 文件类型 |
| preview-support | VARCHAR(20) NOT NULL | 预览支持 | inline_preview/download_only |
| created-at | TIMESTAMPTZ NOT NULL | 创建时间 | 首次上传时间 |
| first-uploader-id | VARCHAR(255) NOT NULL | 首次上传者ID | 用于审计 |

### 索引

- `idx-file-hash-mapping` - 文件哈希索引（去重查询核心）
- `idx-physical-key` - 物理文件key索引

### 去重机制说明

**首次上传**：
1. 计算文件采样SHA-256哈希
2. 查询 `file-hash` 是否存在
3. 不存在则上传文件到MinIO
4. 创建UUID映射记录
5. 在 `file-access-permissions` 表中授予owner权限

**秒传（相同文件）**：
1. 计算文件采样哈希
2. 查询到已存在的UUID映射
3. 为当前用户创建新的 `file-records` 记录
4. 授予当前用户访问权限
5. 返回相同的UUID访问URL
6. **无需重复上传物理文件**

## 📝 数据存储格式说明（仅用于遗留字段）

### TEXT 字段存储约定

> ⚠️ **注意**: 以下格式仅用于 `users` 表和 `groups` 表中尚未迁移的TEXT字段。
> 新功能应使用独立表存储结构化数据。

所有复合数据以**逗号分隔的键值对**形式存储，格式：`key1:value1,key2:value2`

#### 示例：

```
# group-members 字段存储示例：
member-id:user-001,member-nickname:李四,join-time:2025-01-10T08:00:00,member-role:member
```

### 多条记录存储

使用**分号(;)**分隔多条记录：

```
# 多个群成员存储示例：
member-id:user-001,member-nickname:张三,join-time:2025-01-10,member-role:member;member-id:user-002,member-nickname:李四,join-time:2025-01-11,member-role:admin
```

### 查询示例

```sql
-- 查询群聊的所有成员（遗留方式）
SELECT "group-id", "group-name", "group-members" 
FROM "groups" 
WHERE "group-id" = 'group-001';
```

> 💡 **推荐做法**: 将这些TEXT字段迁移到独立表，以获得更好的查询性能和数据完整性。
